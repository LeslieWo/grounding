# Design: Grounding Agent(情感锚定 Agent）

Generated by /office-hours · 2026-07-02
Mode: Builder（个人项目 + agent 学习 + 市场空缺探测）
Status: DRAFT

## 一句话
一个从自己 panic/闪回痛点出发的工具：把美好、安全的记忆照片存进库；闪回来袭时，agent 按「当下情绪状态」而非「语义相关度」捞出最能帮你 ground 回当下的记忆块（视觉/触觉/空间/情绪多轴）。

## 目标（按优先级，这决定所有取舍）
1. **亲身可用** —— 下次 panic 时它真帮到我。
2. **学 agent** —— 建的过程踩到真的 agent 难题（state / retrieval 目标函数 / self-critique / eval）。
3. **探测市场空缺** —— wall-log 记录现成 infra 在这个用例上崩的地方。

明确**不是**目标：盈利、规模、融资。别让它承担这些。

## 为什么它不是又一个 agent memory
所有人的 agent memory 按**语义相关度**检索。这个按**情绪调节效力**检索："什么记忆能让这套神经系统当下平静下来"。这是个不同的目标函数，现成工具（Mem0 / Zep / vector RAG）不解 —— 这就是你的 weird corner，也是潜在的 infra 空缺。

## 约束 / 红线
- **隐私：照片永不出设备。** 这是你最私密的记忆，比 Trace 的脸更敏感。全本地（on-device），LLM 只收「状态描述」文本，不收图。（延续你 Trace「照片不出库」的红线。）
- **安全：检索绝不能吐出会二次触发的照片。** 必须有 safe-set 人工策展 + 硬过滤。
- **load-bearing 工具：** 危机时刻你会依赖它 → 可靠性 > 花哨。而且它是「多一件工具」，不是「唯一一件」，别让它承诺它治不了的东西。
- solo、带宽有限 → 分层建，别一次吞完。

## 前提（先认这几条，再往下）
1. 傻版本先验证「它到底帮不帮我」，帮到了才值得往上长。
2. 学 agent 的价值来自把它建成**多步管线**，不是 CRUD。
3. wall-log 是市场空缺探测器，不是日记；要结构化记录、事后聚类。

## 架构：两层

### Layer 0 — 傻核心（这周能用）
- **grounding 相册：** 手工挑一批「安全、美好」的照片进 safe-set。
- **多轴标签：** 每张打标 —— 视觉（色/光）、触觉（那天摸到什么的质感）、空间（在哪、大/小/开阔）、情绪（平静/被爱/有力量）、人物场景、时间。
- **傻触发：** panic 时打开 → 按标签或随机吐 3-5 张，一次一张，配一句感官锚定提示（"说出你看到的三样东西 / 那天它摸起来是什么感觉"）—— 这句让它是 grounding 工具，不是相册。
- **零摩擦：** 离线、秒开、无登录。panic 时你没法跟 UI 搏斗。
- **一个反馈按钮：** 结束时点一下「有没有帮我回来」（y/n）。这一个 bit 是后面所有学习的燃料。

### Layer 1 — agentic 骨架（学 agent 的层，傻核心跑通后再长）
管线五步：
1. **状态感知 State sensing** —— 输入当下状态（几个 tap：心跳快 / 发麻 / 停不下来；或一句话）。Agent 得先「感知」。
2. **检索策略选择 Policy** —— 不同状态要不同锚：解离→强触觉/感官；思绪狂奔→慢/开阔的空间；悲伤闪回→安全/被爱。这一步是 agentic 决策，不是固定 if-else。
3. **检索 Retrieval** —— 按「预测的调节效力」捞候选，不是按 query 相似度。早期：策略→标签匹配。后期：用反馈学。
4. **自我评估 Self-critique** —— 捞回来这几张：安全吗（不是触发源）？对上目标轴了吗？不对→重试。
5. **反馈闭环 Feedback** —— 事后「帮到了吗」，回灌到检索目标 + wall-log。

**eval 是这里最硬也最值钱的部分：** 「检索有没有真的调节了状态」没有现成 benchmark。你得自己造这个 eval —— 这既是最深的 agent 学习，也是最清晰的空缺信号。（正好接你 Trace ingredient 那套 eval-first 的肌肉。）

## wall-log（市场空缺探测器）
一个结构化表，每条：`{日期, 哪一步崩的, 现成工具, 期望行为, 实际行为, 一句话根因}`。

预判你会撞到的墙（先别信，撞到再记）：
- 现成 embedding 没有「calming-ness」这个维度，vector 检索按语义捞回情绪上错的图。
- 没有现成的「state → retrieval 策略」的 policy 抽象。
- 没有「检索是否调节了情绪状态」的 eval / 数据集。

写满 ~20 条，重复出现的 2-3 堵墙 = 你亲手撞出来的市场空缺。

## 技术选型（建议，你可改）
- **存储/检索：** 本地 SQLite（照片本地）；视觉召回用 CLIP 类多模态 embedding，情感/触觉轴靠结构化标签（因为没有现成 embedding —— 这本身就是 wall #1）。
- **编排：** 复用你含容 agent 的多组件管线架构（安全分类→检索→起草→自我批判，几乎一一对应）。
- **模型：** 状态分类/策略用 API 均可（只传状态文本，不传图）；图不出设备。
- **平台：** 先做你自己天天在用的那个（手机 or 桌面）—— panic 时你伸手就够到的那个。

## 考虑过的其他方案
- **B：只做傻核心，不做 agent。** 最快、真能用，但目标 2、3 全落空（学不到 agent、无空缺信号）。→ 采纳为 Layer 0 里程碑，不作终点。
- **C：第一天就上完整 agentic + eval。** 学得最多，但迟迟没能用的东西，还没验证它帮不帮你就过度工程。→ 违反「傻版本先行」，否。
- **A（推荐）：两层，傻核心先跑 → 长成 agentic 管线。** B 是它的第 0 步，C 的野心分阶段实现。同时喂饱三个目标。

## Success criteria
- **里程碑 0（~7 天）：** Layer 0 能用，且下次 panic 你真的开了它。→ 唯一真正的验证信号（n=1，就是你）。
- **里程碑 1：** 五步管线跑通，state→strategy→retrieval→critique→feedback 全链。
- **里程碑 2：** wall-log ≥ 20 条，聚出 2-3 个重复的墙。
- **里程碑 3（可选）：** 给「检索是否调节状态」搭出一个哪怕很粗的 eval。

## Next steps（建议建造顺序）
1. 挑 30-50 张 safe-set 照片，打多轴标签。（手工，今天就能开始）
2. 搭 Layer 0 傻触发 + 感官提示 + 反馈按钮。这周用起来。
3. 用几次，记 Layer 0 的 wall-log。
4. **帮不到你？停，别往上长（诚实第一）。** 帮到了？加 state sensing + policy，开始长 Layer 1。
5. 逐步补 self-critique 和 eval，wall-log 一路记。

## 我注意到的
- 你说「不为盈利」—— 那是这个决定里最健康的部分。它让你第一次不是为坐标做产品，是为真需求。
- 你两次绕回情绪感知的 agent（含容 agent → 这个）。不是随机游走，是你真正「有名字」的域在冒头。盯着它。
- 你笑我「收入取决于有人崩溃」—— 你是对的，那句是我夹带道德。你能一眼抓出别人论证里的杂质，这个 taste 很值钱。
